# Training Notes Template — 知識傳遞型場次

當會議是「**講者單向傳播 + 學員問答**」(教學、分享、demo 場)時,套這份 template,**不要**套 `notes-template.md`(那份是給 decision-making 會議的)。

## 判斷依據

| 訊號 | 用哪份 template |
|------|---------------|
| 講者花大半時間講課 / 投影片 / 寫白板 | **本份** |
| Demo / Live coding / 概念拆解 | **本份** |
| 學員提問 → 講者回應(Q&A 為主) | **本份** |
| 多人對等討論、達成決議、分配任務 | `notes-template.md` |
| Sprint Planning / Retro / Sync | `notes-template.md` |

混合場(部分教學部分決議)→ 預設用**本份**,把決議放在「關鍵原則」段。

## 寫作規則

### 1. 「TL;DR」vs 「我的 Takeaway」分得開

- **TL;DR** = **講者原意**,只放講者明確說出的概念,寫紀錄時引用講者的話為主
- **我的 Takeaway** = **記錄者觀察**,記錄者跨場次串聯、自己擷取的洞見,**全部加 `_(記錄者推論)_`**

混在一起會讓未來閱讀者誤以為「TL;DR 第 3 點是 Eric 自己這樣講的」,但其實是記錄者推論。

### 2. 概念樹不超過 4 個主概念

每場找出 2-4 個 H3 主概念。**超過 4 個 → 本場分太細 → 通常代表時間軸還沒清乾淨,要再 restructure**。

### 3. 翻車事件單獨記

講者在 demo 時翻車是**最有教學價值的部分**,別藏在「詳細議題」流水帳裡。獨立一個「程式碼 / Demo 範例」段,記下:
- 第 X 次失敗的原因
- 講者當下的判斷 / 解法
- 學員的觀察 / 後續討論

### 4. Q&A 要表格化

不要寫成「然後有學員問 X,Eric 答 Y,然後另一位...」這種敘事文。**表格化 → 未來掃過快**。

### 5. 不要把 decision-meeting 模板硬塞進來

如果這場真的有決議(混合場),把決議併入「關鍵原則」第 N 條,並加引述:「**[5/X 會議當場拍板]** ...」。

**不要**新增「決策清單」H2 段——那會讓本份 template 退化成 decision template,失去「概念樹」結構的價值。

---

## 篇幅控管

| 場次長度 | 紀錄目標長度 |
|----------|-------------|
| < 30 分 | 1 頁(概念樹 2 個主概念) |
| 30-60 分 | 2-3 頁(概念樹 2-3 個主概念) |
| 1-2 小時 | 3-5 頁(概念樹 3-4 個主概念) |
| > 2 小時 | 5-7 頁,必要時拆 |

「程式碼 / Demo」段可以長一點(內容本身就佔篇幅),但 TL;DR + 核心概念樹要讓讀者 1 分鐘抓到重點。

---

## 反例(不要這樣寫)

❌ **「詳細議題 #1、#2、#3」按時間順序**:這就是 decision-template 的特徵,training notes 該用概念樹
❌ **TL;DR 混入記錄者推論**:破壞「講者原意 vs 觀察者」分離
❌ **Q&A 寫成敘事段**:失去查詢價值
❌ **5+ 個 H3 主概念**:概念樹過散
❌ **每場都列完整 glossary**:對照表只列本場 actual hit 的子集,完整版見 glossary.md
