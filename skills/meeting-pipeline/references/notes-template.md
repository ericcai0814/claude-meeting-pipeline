## 寫作規則

### 1. 保留不確定性

Whisper 對英文人名 / 技術名詞的中文化偏移率高。**不要強裝確定**。

| 情境 | 做法 |
|------|------|
| 完全不確定的人名 | `_(?)_` 註腳 |
| 推測但有偏移痕跡的技術名詞 | `Naple _(Nebula?)_` |
| 完全聽不出 | `_<unclear>_` |

範例（這次會議實例）：
- 我把 `Aloba` 誤判為公司名 → 實際是 `Arova`
- 把 `Nebula` 寫成 `Nepal/Naple` → 實際是 `Netbird`
- 把 `Cloud Design` 寫成 `Cloud` → 實際是 `Claude Design`

→ 若沒留 `(?)` 標記，校稿者無法快速找出哪些要回頭聽 srt 確認。

### 2. Owner 認定原則

**Whisper 不會明確告訴你誰說了什麼**（除非用 whisperx 加 diarization）。所以：

- 從脈絡推測（「我來做 X」「Eric 你那邊...」）
- 推測不確定時用 `_(?)_`
- **拆 issues 前必須 user 校正過 owner**——這是流程強制人工檢查點

### 3. 區分「會議當場已完成」vs「待辦」

- `[x]` = 會議當場決議或執行完
- `[ ]` = 會後待辦

範例：Kyle 在會議中即時做 `Aloba OPS repo 初始化`、`補 infra 驗證腳本` → 會議紀錄裡這兩項應該標 `[x]` 而非 `[ ]`。

### 4. Action Items 分組策略

**by 人分組**而不是 by 領域：

```markdown
### Eric
- [ ] ...
- [ ] ...

### Kyle
- [ ] ...
```

不要：

```markdown
### Infra
- [ ] Eric: ...
- [ ] Kyle: ...

### Hackathon
- [ ] Eric: ...
```

理由：每個 owner 主要在意「我要做什麼」，by 人分組讓他們不用 grep 自己名字。

### 5. 風險顧慮獨立區段

風險不是 action item，但會議常常產生「這件事可能會出問題」的觀察。獨立列出：

- 不需要立即行動，但決策時要記得
- 後人讀紀錄能快速看到顧慮點

範例：
- 「客戶 by case tune 成本爆炸」是一個風險觀察
- 「業務推得太快可能損及規格品質」是另一個

## 連結回原始檔的規則

每份會議紀錄頂部必須 link 回：

1. `.m4a` 原音訊（用 `[name](relative-path)（LFS）` 標註）
2. `.txt` 純逐字稿
3. `.srt` 時間戳版本（**必須**——這是「跳段重聽」的入口）

理由：會議紀錄是壓縮版，**支援證據在 srt 裡**。對某段紀錄有疑義時，能直接用時間戳跳到原音訊驗證。

---

## 篇幅控管

| 會議長度 | 紀錄目標長度 |
|----------|-------------|
| < 30 分 | < 1 頁 |
| 30-60 分 | 1-2 頁 |
| 1-2 小時 | 2-4 頁 |
| > 2 小時 | 4-6 頁，必要時拆主題 |

詳細議題部分可以更長，但 TL;DR + 決策 + Action Items 三個區段加起來應該能讓讀者 30 秒抓到重點。

---

## 反例（不要這樣寫）

❌ **逐字謄錄成長段對話**：失去整理的價值
❌ **Action items 沒有 owner**：拆不出 issue
❌ **強制每個議題都寫完整三段（背景/討論/結論）**：很多議題只有結論值得記
❌ **不留 `(?)` 標記**：校稿者不知道哪裡要回頭聽
❌ **混淆 `[x]` 和 `[ ]`**：當場做完跟會後待辦差很多
